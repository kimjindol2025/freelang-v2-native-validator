# FreeLang v2 - 헤더 계약 시스템

## 🎯 핵심 원칙

**"헤더는 계약이면서 코딩의 전체입니다"**

```
사과 >> 먹다 >> 없다
 ↑      ↑      ↑
시작    중간    결과
```

---

## 📜 헤더 = 계약

### 헤더 정의 (필수)

```freelang
fn 함수명: 입력 → 출력 ~ "의도"
```

**예시**:
```freelang
fn sum: array<number> → number ~ "배열의 모든 수를 더해서 합을 반환한다"

fn average: array<number> → number ~ "배열의 평균을 계산한다"

fn filter: array<number>, threshold: number → array<number> ~ "threshold 이상의 값만 필터링한다"
```

---

## 🔗 헤더의 세 가지 역할

### 1️⃣ 시작 (입력 정의)
```
array<number> ← 이것을 받아서
```

### 2️⃣ 중간 (의도/계약)
```
"배열의 모든 수를 더한다" ← 이것을 한다
```

### 3️⃣ 결과 (출력 정의)
```
number → 이것을 반환한다
```

**헤더가 정의하면 나머지는 자동**:
- 입력 검증 자동
- 출력 타입 자동
- 의도 검증 자동

---

## 💻 중간 코드 = 자유로움

### 규칙
1. 필기체 OK (문법 엄격하지 않음)
2. 부분 완성 OK (AI가 채움)
3. 여러 방식 OK (for, while, 뭐든)

### 예시

**완벽한 코드**:
```freelang
fn sum: array<number> → number ~ "배열 합산"
  result = 0
  for i in array
    result += i
  return result
```

**필기체 코드**:
```freelang
fn sum: array<number> → number ~ "배열 합산"
  r = 0
  loop arr -> r += it
  r
```

**부분 완성**:
```freelang
fn sum: array<number> → number ~ "배열 합산"
  result = 0
  for i in array
    // 비움 (AI가 채움)
  return result
```

**모두 같은 결과를 만듭니다!**

## ✅ 검증 = 헤더 만족

### 검증 프로세스

1. **헤더 명확성 검사**
   - 입력 타입 정의됨?
   - 출력 타입 정의됨?
   - 의도 명확함?

2. **의도 테스트**
   - 자동 테스트 케이스 실행
   - 예: `sum([1,2,3]) == 6?`

3. **결과 확인**
   - 헤더 의도 만족?
   - 타입 매칭?

### 통과 조건

```
헤더 명확? ✅
테스트 통과? ✅
→ 컴파일 성공!
```

## ⚠️ 컴파일 오류 메시지

### Error 1: 헤더 미정의

```
❌ ERROR: 헤더 정의 미흡

  fn sum: ??? → ??? ~ "???"
          ^^^   ^^^   ^^^

메시지:
  "헤더는 계약입니다.

   사과 >> 먹다 >> 없다
   시작  중간  결과

   형식: fn 함수명: 입력 → 출력 ~ "의도"

   헤더 문장을 정확하게 작성하시오.
   다음 코드 작성 시 AI가 패턴을 학습합니다."
```

### Error 2: 입력 타입 미정의

```
❌ ERROR: 입력 타입 미정의

  fn sum: ??? → number ~ "배열 합산"
          ^^^

메시지:
  "입력 타입을 명확히 하세요.

   예: array<number>, string, int, boolean

   헤더가 명확하면 컴파일이 진행됩니다."
```

### Error 3: 의도 미정의

```
❌ ERROR: 의도 미정의

  fn sum: array<number> → number ~ "???"
                                    ^^^

메시지:
  "의도를 자연어로 명확히 작성하세요.

   예: "배열의 모든 수를 더해서 합을 반환한다"

   의도가 명확하면 AI가 자동 검증합니다."
```

### Error 4: 의도 불만족

```
❌ ERROR: 컴파일 오류 - 의도 불만족

  fn sum: array<number> → number ~ "배열 합산"

테스트 실패:
  sum([1,2,3]) expected: 6, actual: 5 ❌

메시지:
  "헤더 의도를 만족하지 않습니다.

   의도: "배열의 모든 수를 더한다"
   테스트: sum([1,2,3]) == 6

   중간 코드를 검토하고 수정하세요."
```

## 🧠 AI 패턴 인식 & 학습

### 메커니즘

```
1차 시도: 헤더 미흡 오류
  ↓
AI 패턴 인식
  "아, 헤더는 이렇게 명확하게 써야 하는구나"
  ↓
2차 시도: 명확한 헤더 작성
  fn sum: array<number> → number ~ "배열의 모든 수를 더해서 합을 반환한다"
  ↓
컴파일 성공!
```

### 예시

**첫 번째 코드** (AI가 작성):
```freelang
fn sum: array ~ "더하기"
// ❌ 헤더 미흡 오류
```

**오류 메시지 받은 후**:
```freelang
// AI 학습: "헤더는 입력 → 출력 ~ 의도여야 한다"
```

**두 번째 코드** (AI가 학습):
```freelang
fn sum: array<number> → number ~ "배열의 모든 수를 더해서 합을 반환한다"
// ✅ 컴파일 성공!
```

## 📋 헤더 작성 체크리스트

### 필수 요소

- [ ] 함수명 명확?
- [ ] 입력 타입 명시? (array<T>, int, string 등)
- [ ] 출력 타입 명시?
- [ ] 의도 명확? (자연어 문장)
- [ ] ">> 구조" 이해? (시작 >> 중간 >> 결과)

### 좋은 헤더 예시

```freelang
fn sum: array<number> → number
  ~ "배열의 모든 요소를 더해서 합을 반환한다"

fn filter: array<number>, threshold: number → array<number>
  ~ "배열에서 threshold 이상의 값만 필터링해서 새 배열을 반환한다"

fn max: array<number> → number
  ~ "배열의 최대값을 찾아서 반환한다"
```

### 나쁜 헤더 예시

```freelang
❌ fn sum: array ~ "더하기"
   (입력 상세 미흡, 출력 없음, 의도 모호)

❌ fn filter: array → array ~ "필터"
   (threshold 파라미터 누락, 의도 모호)

❌ fn max: array → number
   (의도 없음)
```

## 📝 언어 명시

**FreeLang v2 선언**:

```
이 언어에서 "헤더"는 단순한 함수 시그니처가 아닙니다.

헤더는 "계약"이면서 "코딩의 전체"입니다.

사과 >> 먹다 >> 없다
(시작) (중간) (결과)

- 시작: 무엇을 받을 것인가? (입력 타입)
- 중간: 무엇을 할 것인가? (의도/목적)
- 결과: 무엇을 반환할 것인가? (출력 타입)

헤더가 명확하면, 중간 코드는 자유롭습니다.
필기체든, 부분 완성이든, 어떤 형태든 상관없습니다.

AI가 헤더를 읽고 의도를 이해합니다.
AI가 중간 코드를 보정합니다.
AI가 헤더 의도를 만족하는지 검증합니다.

컴파일 오류가 나면, 그것은 학습입니다.
패턴을 인식하고, 다음 코드는 더 명확하게 작성합니다.

이것이 AI-First 언어의 철학입니다.
```
